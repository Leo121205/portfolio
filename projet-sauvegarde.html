Plan de sauvegarde 3-2-1

Mise en place d'une politique de sauvegarde des données critiques avec Azure Backup, scripts Robocopy et procédures de restauration testées.

Contexte & Objectifs

La perte de données est l'un des risques majeurs pour les entreprises. Ce projet met en œuvre la règle 3-2-1 de sauvegarde : 3 copies des données, sur 2 supports différents, dont 1 copie hors site (cloud Azure). L'environnement s'appuie sur l'infrastructure Active Directory efrei.local déjà en place.

Règle 3-2-1 appliquée

  • Copie 1 : Données source sur le serveur de fichiers (SRV-FILES)
  • Copie 2 : Sauvegarde locale sur NAS virtuel (scripts Robocopy planifiés)
  • Copie 3 : Sauvegarde hors site sur Azure Backup (coffre Recovery Services)

Données critiques identifiées

  • Dossiers partagés réseau (\\SRV-FILES\Partage$) — 15 Go
  • Base de données GLPI (dump MariaDB quotidien) — 500 Mo
  • État du système Active Directory (System State Backup) — 8 Go

Étapes de réalisation

  1. Analyse des besoins et politique de sauvegarde

    Identification des données critiques, classification par niveau de sensibilité. Définition des RTO (Recovery Time Objective : 4h) et RPO (Recovery Point Objective : 24h). Documentation de la politique dans un Plan de Reprise d'Activité (PRA).

  2. Script Robocopy — Sauvegarde locale quotidienne

    Script PowerShell avec Robocopy : copie incrémentielle des dossiers partagés vers le NAS (D:\Backups\). Options : /MIR (miroir), /LOG (journalisation), /R:3 (3 tentatives), /W:10 (attente 10s entre tentatives). Rétention : 7 jours de sauvegardes incrémentielles.

  3. Planification via le Planificateur de tâches Windows

    Tâche planifiée : exécution tous les jours à 02h00 (heure creuse). Compte de service dédié avec droits de lecture sur la source et écriture sur la destination NAS. Notification par email en cas d'échec (via SMTP Windows).

  4. Sauvegarde système — Windows Server Backup

    Installation du rôle Windows Server Backup. Sauvegarde complète hebdomadaire (dimanche 03h00) incluant le System State (état AD, registre, fichiers système). Sauvegarde vers lecteur dédié \\NAS\SystemBackup$.

  5. Configuration Azure Backup

    Création d'un coffre Azure Recovery Services. Installation de l'agent MARS (Microsoft Azure Recovery Services) sur le serveur. Planification : sauvegarde quotidienne à 03h30, rétention 30 jours. Chiffrement des données en transit et au repos (AES-256).

  6. Tests de restauration

    Test 1 — Restauration de fichier individuel : restauration d'un fichier supprimé depuis la sauvegarde Robocopy (succès, 2 minutes). Test 2 — Restauration complète d'un dossier depuis Azure Backup (succès, 8 minutes). Test 3 — Restauration de l'état AD sur un serveur secondaire (succès, procédure documentée).

Résultats

3Copies garanties
2 minRestauration fichier
4hRTO garanti
24hRPO garanti

Les trois tests de restauration ont été réalisés avec succès et les durées mesurées sont conformes aux objectifs RTO/RPO définis. Les journaux Robocopy ne signalent aucune erreur sur 14 jours de tests consécutifs.

Ce que j'ai appris

Concepts de continuité d'activité : RTO, RPO, PRA, PCA

Scripting PowerShell avancé avec Robocopy pour l'automatisation des sauvegardes

Configuration d'Azure Backup et de l'agent MARS pour la sauvegarde cloud

Importance de tester régulièrement les procédures de restauration
